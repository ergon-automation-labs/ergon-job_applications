# Job Application Ranking Algorithm

## Overview

The Job Application Ranking module implements a composite scoring algorithm to rank job applications by actionability and fit. Applications are scored on a 0-1 scale based on four factors:

1. **Coverage Score** (40%) — How well your resume matches the job description
2. **State Priority** (30%) — Which pipeline stage is most actionable now
3. **Salary Range** (20%) — Whether compensation is acceptable
4. **Role Alignment** (10%) — Whether the role matches your experience level

## Quick Start

```elixir
alias BotArmyJobApplications.{ApplicationStore, Ranking}

# Get all applications
applications = ApplicationStore.list()

# Rank them (highest score first)
ranked = Ranking.rank(applications)

# Get top 5 most actionable applications
top_5 = Ranking.top_n(applications, 5)

# Group by tier (high/medium/low priority)
{high, medium, low} = Ranking.by_tier(applications)
```

## Scoring Algorithm

### Coverage Score (40% weight)

- **Input:** `coverage_score` field (0-1 float or 0-100 integer)
- **Logic:** Direct measurement of how well your resume bullets cover the JD's requirements
- **Range:** 0.0 (no match) to 1.0 (perfect match)
- **Generated by:** `ArtifactWorker` during artifact generation phase

Example: Resume has 5/7 required skills → coverage_score = 0.71

### State Priority (30% weight)

Prioritizes applications that are ready to act on now:

| State | Priority | Weight |
|-------|----------|--------|
| `ready_to_submit` | 🔴 Highest | 1.00 |
| `drafting` | | 0.95 |
| `identified` | 📌 Current | 0.90 |
| `submitted` | | 0.70 |
| `phone_screen` | | 0.60 |
| `technical` | | 0.50 |
| `offer` | | 0.40 |
| `accepted` | ✅ Done | 0.30 |
| `declined`, `rejected`, `ghosted` | ❌ Final | 0.00 |

**Why?** Applications ready to submit are actionable today. In-process applications are good signals but don't need immediate action.

### Salary Range (20% weight)

Evaluates compensation expectations:

| Range | Score |
|-------|-------|
| min ≥ $150k, max ≥ $200k | 1.00 |
| min ≥ $120k, max ≥ $150k | 0.90 |
| min ≥ $100k, max ≥ $120k | 0.70 |
| min ≥ $80k, max ≥ $100k | 0.50 |
| Below $80k min | 0.30 |
| Missing `salary_range` | 0.50 (neutral) |

**Why?** Target roles align with compensation expectations. High salary roles are often more competitive or specialized.

### Role Alignment (10% weight)

Based on `jd_tags` extracted by LLM analysis:

**Seniority Match:**
- Lead/Staff: 0.95
- Senior: 0.90
- Mid: 0.80
- Junior: 0.60
- Entry: 0.50
- Unknown: 0.50

**Technology Specificity:**
- 8+ technologies/frameworks: 1.00
- 5-7: 0.90
- 3-4: 0.70
- 1-2: 0.50
- 0: 0.30

**Combined:** Average of seniority + specificity scores

**Why?** Specific, targeted roles (with seniority match) indicate good fit. Generic roles with no tech requirements are harder to tailor.

## Usage Examples

### Rank All Applications

```elixir
ranked = Ranking.rank(ApplicationStore.list())
# => [
#   {%{id: "...", company: "Google", coverage_score: 0.92, ...}, 0.91},
#   {%{id: "...", company: "Meta", coverage_score: 0.78, ...}, 0.85},
#   {%{id: "...", company: "Startup", coverage_score: 0.55, ...}, 0.68}
# ]
```

### Get Top N Applications

```elixir
top_10 = Ranking.top_n(ApplicationStore.list(), 10)

# Use for "Top 10 most actionable applications right now"
```

### Group by Priority Tier

```elixir
{high, medium, low} = Ranking.by_tier(ApplicationStore.list())

# Tiers:
# high: score >= 0.75 (do today)
# medium: 0.50 <= score < 0.75 (do this week)
# low: score < 0.50 (revisit later)
```

### Single Application Score

```elixir
application = ApplicationStore.get(app_id)
score = Ranking.score(application)
# => 0.84

if score >= 0.75 do
  # High priority — send application today
else
  # Medium priority — gather more info first
end
```

### Debug Scoring

```elixir
app = ApplicationStore.get(app_id)

# Check each factor:
coverage = (app["coverage_score"] || 0.5) |> min(1.0) |> max(0.0)
state_weight = %{
  "ready_to_submit" => 1.0,
  "submitted" => 0.70,
  # ... full map in ranking.ex
}[app["state"] || "identified"]

IO.inspect(%{
  coverage: coverage,
  state: state_weight,
  salary: app["salary_range"],
  jd_tags: app["jd_tags"],
  final_score: Ranking.score(app)
})
```

## Integration with ApplicationStore

The `ApplicationStore` provides the `list()` operation that `Ranking` consumes:

```elixir
def show_actionable(limit \\ 10) do
  ApplicationStore.list()
  |> Ranking.top_n(limit)
  |> Enum.each(fn {app, score} ->
    IO.puts("#{app["company"]} — #{app["role_title"]} (#{Float.round(score, 2)})")
  end)
end
```

## NATS Integration (Future)

When TUI or dashboard requests ranked applications, the pattern will be:

```
request: job.applications.rank {limit: 10}
response: events.job.applications.ranked [
  {id: "...", score: 0.92},
  {id: "...", score: 0.85},
  ...
]
```

Handler implementation:
```elixir
def handle_rank_request(payload) do
  limit = Map.get(payload, "limit", 10)

  ranked = ApplicationStore.list()
    |> Ranking.top_n(limit)
    |> Enum.map(fn {app, score} ->
      %{id: app["id"], score: score}
    end)

  Publisher.publish("events.job.applications.ranked", ranked)
end
```

## Future Enhancements

Phase 2+ could add:

- **Historical Performance:** Track which applications → offers. Re-weight factors by outcome.
- **Location Scoring:** Prefer roles in certain cities/remote arrangements.
- **Company Signals:** Extract from `jd_tags`: funding stage, headcount, glassdoor rating.
- **Skill Recency:** Boost roles using technologies you've used recently.
- **Interview Performance Correlation:** After 50 applications, identify what leads to interviews.
- **Declining Interest:** Lower score if application age > 30 days without action.

## Edge Cases Handled

| Scenario | Behavior |
|----------|----------|
| `coverage_score` > 1.0 | Clamped to 1.0 |
| `coverage_score` < 0.0 | Clamped to 0.0 |
| Missing `coverage_score` | Defaults to 0.5 (neutral) |
| Missing `salary_range` | Defaults to 0.5 (neutral) |
| Missing `state` | Defaults to "identified" (0.9) |
| Missing `jd_tags` | Defaults to 0.5 (neutral) |
| Empty list | Returns `[]` |
| N > list length | Returns all applications |
| N = 0 | Returns `[]` |
